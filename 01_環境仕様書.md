# 環境仕様書（ディレクトリ構成）

本書は spec-driven-development リポジトリの **ディレクトリ構成・各ファイルの役割・構成の設計思想** を説明するドキュメントです。

> **ドキュメント体系**: 本リポジトリのドキュメントは 4 分冊で構成されます。
> ① **環境仕様書**（本書: 構成と設計思想）→ ② **導入手順書**（初期セットアップ）→ ③ **利用手順書**（開発の流れ）→ ④ **運用手順書**（メンテナンス手順）

---

## 1. リポジトリの目的

CMS インフラ構築サービス向けの **仕様駆動開発** リポジトリです。Kiro を用いて、QA 票・基本設計書・構成図などのインプットから「要件定義 → 基本設計書 → Terraform コード」を半自動生成するための **共通設定（ルール・スキル）** と、参照用の既存 Terraform リポジトリ群（Git Submodule）を集約しています。

- チーム全員で同じ Kiro 設定を共有し、**個人のプロンプト作成力に依存しない**生成品質を実現する
- 各 CSP（`aws` / `azure` / `gcp`）の既存モジュールを Submodule として参照し、**成果物の品質を揃える**
- スキルは **コマンドで明示起動**（LLM によるパターン自動判定は行わない）し、動作の再現性を担保する

## 2. ディレクトリ構成（全体ツリー）

各ファイルの詳細は §3 の表、配置の考え方は §4 の設計思想を参照してください。

```
spec-driven-development /
├── .kiro/                              # Kiro 共通設定 ★Git 管理
│   ├── steering/                       # 常時/条件付きで適用されるルール群
│   │   ├── coding-conventions.md
│   │   ├── naming-conventions.md
│   │   ├── dcs-env-definitions.md
│   │   └── language-preferences.md
│   ├── agents/                         # 【予約枠】カスタムエージェント配置用（未導入）
│   ├── hooks/                          # 【予約枠】自動化フック配置用（未導入）
│   ├── settings/
│   │   └── mcp.example.json            # MCP 設定テンプレート（実動 mcp.json は各自作成）
│   └── skills/                         # 明示起動する生成スキル群
│       ├── generate-design/            # 仕様書・設計書の生成
│       │   ├── SKILL.md
│       │   ├── parts/                  #   aws.md / azure.md / gcp.md
│       │   └── references/             #   {cloud}-doc-format.md
│       ├── generate-code/              # Terraform コードの生成
│       │   ├── SKILL.md
│       │   ├── parts/                  #   aws.md / azure.md / gcp.md
│       │   └── references/             #   {cloud}-module-catalog.md
│       └── generate-diagram/           # 構成図（draw.io）の生成
│           └── SKILL.md
├── EngRepos/                           # 参照用の外部 Terraform リポジトリ群（Git Submodule）★Git 管理
├── Outputs/                            # 案件ごとの成果物（要件・設計書・構成図・コード）🚫各自ローカル
├── AGENTS.md                           # 生成 AI 向けの目次
├── README.md
├── .kiroignore / .gitignore / .gitmodules
```

## 3. 主要ファイル詳細

メンテナンスで更新が発生しやすい主要ファイルについて、役割・読み込まれ方・更新頻度の目安を示します。

| ファイル | 役割（何が書いてあるか） | 読み込まれ方 | 更新頻度 |
|---|---|---|---|
| `steering/coding-conventions.md` | Terraform コーディング規約の**正**。チーム固有の決定事項（ファイル命名・バージョン標準・backend ルール等）のみ記載し、一般的な作法は HashiCorp 公式へ委譲 | fileMatch（`**/*.tf` を扱うときだけ自動適用） | 中（規約改定時） |
| `steering/naming-conventions.md` | 案件フォルダ名・出力先パスなど**命名の正** | always（全対話で常時適用） | 中 |
| `steering/dcs-env-definitions.md` | DCS Hub 環境の事実（NW・払い出し範囲）と CMS 標準動作（SG/NSG・バックアップ・暗号化等）の**正** | manual（generate-design / generate-code の SKILL.md が起動時に必読指示） | 低〜中（環境仕様変更時） |
| `steering/language-preferences.md` | 全出力を日本語に統一する言語設定 | always | 低 |
| `skills/generate-*/SKILL.md` | **各スキルの起動時に Kiro が最初に読む手順書**。「引数（案件フォルダ名）の受け取り方」「先に読むべき steering」「どの順で何を生成するか」「どこに出力するか（パス直指定）」「人間承認で停止するタイミング」が書かれている | `/generate-*` コマンド起動時のみ | 中（フロー改善時） |
| `skills/*/parts/{cloud}.md` | CSP 別の記入ルール・リソース略称。**案件の CSP に該当する 1 ファイルだけ**が読み込まれる | SKILL.md からの指示で必要分のみ | 中（CSP 知見の蓄積で追記） |
| `skills/generate-design/references/{cloud}-doc-format.md` | 設計書の章立て・表形式の定義（設計書フォーマットの正） | SKILL.md からの指示で必要時のみ | 低 |
| `skills/generate-code/references/{cloud}-module-catalog.md` | 利用可能モジュールの一覧と EngRepos 内の参照パス（詳細仕様は転記せず EngRepos を正とする） | 同上 | 中（モジュール追加時） |
| `settings/mcp.example.json` | MCP 接続設定のテンプレート。各自がコピーして `mcp.json`（Git 管理外）を作成する | —（人間用） | 低 |
| `AGENTS.md` | 生成 AI 向けの目次。基本原則・スキル一覧・steering 一覧を要点のみ記載（詳細は各ファイルが正） | Kiro が自動参照 | 低（構成変更時に追従） |
| `EngRepos/` | 各 CSP の既存 Terraform モジュール（Submodule・特定コミットに固定） | generate-code がカタログ経由で参照 | 中（ポインタ更新） |

## 4. 構成の設計思想（なぜこの配置なのか）

本リポジトリの構成は、Kiro および生成 AI エージェント運用のベストプラクティスに基づき、以下の 5 つの思想で設計されています。**ファイルの追加・移動・更新はこの思想に沿って行ってください**（詳細な変更手順は運用手順書を参照）。

### 思想①: steering と skills の分離 —「常に守るルール」と「呼ばれたら動く手順」

- **steering** には「宣言的なルール・環境の事実」だけを置きます。Kiro の steering は inclusion（always / fileMatch / manual）によって**自動で文脈に読み込まれる**仕組みであり、常時守らせたい規約に適しているためです。
- **skills** には「特定タスクの実行手順（手続き）」を置きます。スキルは Agent Skills のオープン標準に沿った**オンデマンド読み込み**であり、必要なときだけ文脈に載るため、クレジット消費と文脈汚染を最小化できます。
- **配置の判断基準**: 「すべての作業で常に効いてほしいか？」→ Yes なら steering。「特定の生成タスクのときだけ必要か？」→ Yes なら skills。

### 思想②: 段階的読み込み（progressive disclosure）— skills 内が 3 層である理由

`SKILL.md`（入口・薄い）→ `parts/{cloud}.md`（案件 CSP の分だけ）→ `references/`（必要時のみ）という 3 層構造は、**一度に読み込む量を最小にする**ための設計です。たとえば AWS 案件で Azure/GCP のルールを読み込むのは無駄であるだけでなく、無関係な情報が生成品質を下げる要因になります。このため「他 CSP の parts は読まない」ことをスキル側で明示しています。

### 思想③: 正の一元化（single source of truth / DRY）

同じ情報は**必ず 1 ファイルにだけ**書きます。例: Terraform 規約は `coding-conventions.md`、命名は `naming-conventions.md`、出力先パスは各 `SKILL.md`、モジュール仕様は EngRepos の実装。他ファイルは参照するのみとし、転記しません。転記すると更新時の同期漏れが必ず発生するためです。

### 思想④: 明示起動 — LLM の自動判定を挟まない

スキルはコマンド（`/generate-*` ＋ 案件フォルダ名）で明示起動します。指示文から LLM がパターンを推測する構成は、誤判定→対症療法の指示追加→運用の複雑化を招くため排除しています。この思想により、steering に「ルーター役」のファイルは存在しません。

### 思想⑤: 共有物と個人・案件固有物の分離

- チームで共有すべき設定（ルール・スキル・テンプレート）は Git 管理し、**個人環境に依存する設定**（`mcp.json`: バイナリパスが人ごとに異なる）と**案件固有の生成物**（`Outputs/`）は Git 管理外とします。Kiro の成果物（要件・設計書・構成図・コード）はすべて `Outputs/` 配下に統一しています。
- `agents/` `hooks/` は「定義は可能だが未導入」という状態を構成上明示するための**予約枠**です（`.gitkeep` のみ）。

> **⚠️ メンテナンス時の遵守事項**: ファイルの更新・追加の際は上記思想に乗っ取ってください。**もし思想に反する更新（例: steering に手続きを書く、情報を複数箇所へ転記する）がどうしても必要になった場合は、本書のこの章の記載も併せて変更してください**。実態と仕様書の乖離が最も危険です。

## 5. Git 管理対象

| 区分 | 対象 | 説明 |
|---|---|---|
| ✅ Git 管理（チーム共有） | `.kiro/` | 共通ルール・スキル。ただし `settings/` は `mcp.example.json` のみ共有 |
| ✅ Git 管理（チーム共有） | `EngRepos/` `.gitmodules` | Submodule 本体と定義 |
| ✅ Git 管理（チーム共有） | `README.md` `AGENTS.md` `.gitignore` `.kiroignore` | ドキュメント・無視ルール |
| 🔒 各自ローカル（Git 管理外） | `.kiro/settings/mcp.json` | MCP の実動設定。バイナリのパスが人ごとに異なるため各自作成（導入手順書参照） |
| 🔒 各自ローカル（Git 管理外） | `Outputs/` | 案件ごとの成果物（要件・設計書・構成図・コード）。共有が必要なものは各案件のリポジトリ／保管先で別途管理 |
