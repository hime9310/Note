# 導入手順書（初期セットアップ）

本書は Kiro を用いた仕様駆動開発を始めるための **初回セットアップ手順**（所要 20〜30 分）です。上から順に実施し、各ステップの「✔ 確認」ができたら次へ進んでください。

> **ドキュメント体系**: ① 環境仕様書（構成と設計思想）→ ② **導入手順書（本書）** → ③ 利用手順書（開発の流れ）→ ④ 運用手順書（メンテナンス手順）

---

## 0. 前提: ソフトウェアのインストール状況確認

本手順は、以下の管理台帳に記載のソフトウェアが**インストール済みであること**を前提とします。**未実施の場合は、先に台帳に従って以下をインストールしてください。**

📋 **[IaC 利用ソフトウェア管理台帳.xlsx](＜台帳の格納先リンクに差し替え＞)**

| 区分 | ツール | 用途 | 確認コマンド |
|---|---|---|---|
| **必須** | Node.js（**v18 以上**） | draw.io MCP の実行 | `node -v` |
| **必須** | npm / npx | draw.io MCP のインストール・起動 | `npx -v` |
| **必須** | Git（2.13 以上） | クローン・Submodule 管理 | `git --version` |
| **必須** | Git Credential Manager | Azure DevOps 認証（PAT 推奨） | — |
| **必須** | Kiro（**1.0 以上**） | 仕様駆動のドキュメント／コード生成 IDE | —（Step 1 で導入） |
| **必須** | Python（**3.9 以上**） | Excel 解析・hook スクリプトの実行基盤 | `python --version` |
| **必須** | openpyxl | Excel の読み書き（QA 票の解析・設計書 Excel 出力） | `pip show openpyxl` |
| **必須** | pandas | Excel データの表形式処理 | `pip show pandas` |
| 推奨 | Terraform | 生成コードの検証（`fmt` / `validate`） | `terraform -v` |
| 任意 | terraform-docs | 成果物 README の自動生成 | `terraform-docs --version` |

```powershell
# PowerShell での確認例
node -v      # v18 以上ならOK（例: v24.11.1）
npx -v       # バージョンが表示されればOK
python --version                                   # 3.9 以上
python -c "import openpyxl, pandas; print('OK')"   # OK が出れば導入済み
```

```powershell
# Python ライブラリ未インストールの場合
pip install openpyxl pandas
```

> - Node.js / npm（npx）は **draw.io MCP の実行・インストールに必須**です。未導入の場合、構成図生成（`/generate-diagram`）が動作しません。
> - Python / openpyxl / pandas は **Excel の解析・出力に必須**です（QA 票の読み込み、および `/generate-design` の設計書 Excel 同時出力に使用）。未導入の場合、設計書生成が失敗します。

**✔ 確認**: 必須ツールがすべて確認コマンドに応答し、`import openpyxl, pandas` が `OK` を返す。

## Step 1. Kiro のインストールと初期ログイン

1. Kiro 公式サイト **https://kiro.dev/** にアクセスし、インストーラをダウンロードして実行する
   - **基本方針として最新バージョンを使用**します。ただし **1.0 リリース以降、自動アップデートは停止中**（再開時期未定）のため、インストーラ経由で取得したバージョンのまま利用します（バージョン更新の運用は 04 運用手順書参照）
2. Kiro を起動し、サインイン方式で **AWS IAM Identity Center** を選択して初期ログインする（チーム標準の認証方式。個人の AWS Builder ID 等は使用しない）
3. 使用モデルが **Claude Opus 4.7（チーム標準・デフォルト設定済み）** になっていることを確認する（モデル方針は 03 利用手順書参照）

**✔ 確認**: Kiro が起動し、IAM Identity Center でログイン済み・モデルが Claude Opus 4.7 になっている。

## Step 2. リポジトリのクローン（Submodule 込み）

```bash
git clone --recurse-submodules https://JPCMS@dev.azure.com/JPCMS/CMS_Eng_Common/_git/cms-eng-spec-driven-development
cd cms-eng-spec-driven-development
```

`--recurse-submodules` を付け忘れた場合は、続けて次を実行します。

```bash
git submodule update --init --recursive
```

> Submodule は Azure DevOps（`dev.azure.com/JPCMS`）にホストされています。各 CSP プロジェクト（`CMS_AWS` / `CMS_Azure` / `CMS_GCP`）へのアクセス権が必要です。認証エラーになる場合はアクセス権・PAT を確認してください。

**✔ 確認**: `eng-repos/` 配下にモジュール群が展開されている（空なら上記コマンドを実行）。

## Step 3. Terraform MCP Server の配置

`.tf` 生成時に公式仕様を参照するための MCP サーバーを配置します（コーディング規約上、利用必須）。

1. **https://releases.hashicorp.com/terraform-mcp-server/** から**最新バージョン**のバイナリをダウンロードする
   - Windows（64bit）の場合: `terraform-mcp-server_X.X.X_windows_amd64.zip`
2. 解凍後、バイナリを任意のパスに配置する
   - 配置例: `C:\Users\<ユーザー名>\mcp\terraform-mcp-server\terraform-mcp-server.exe`
3. コマンドプロンプトで起動確認する（エラーなく起動すれば OK。`Ctrl+C` で終了）

```
C:\Users\<ユーザー名>\mcp\terraform-mcp-server\terraform-mcp-server.exe stdio
```

**✔ 確認**: `Terraform MCP Server running on stdio` が表示される。
（`TFE client` 関連の warning は Terraform Cloud 未使用のため無視して問題ありません）

## Step 4. MCP 設定ファイルの作成（mcp.example.json をコピー）

`mcp.json` は**各自のローカル設定**です（Git 管理外。バイナリのパスが人ごとに異なるため）。テンプレートからコピーして作成します。

1. `.kiro/settings/mcp.example.json` を**同じフォルダに**コピーし、`mcp.json` にリネームする
2. `terraform` の `command` のパスを、**Step 3 で自分が配置した場所に合わせて変更**する

```jsonc
// .kiro/settings/mcp.json（例）
{
  "mcpServers": {
    "terraform": {
      "command": "C:\\Users\\<ユーザー名>\\mcp\\terraform-mcp-server\\terraform-mcp-server.exe",  // ★自分のパスに変更
      "args": ["stdio"],
      "disabled": false,
      "autoApprove": [
        "get_latest_provider_version",
        "search_providers",
        "get_provider_details"
      ]
    },
    "drawio": {
      "command": "npx",
      "args": ["@drawio/mcp"],
      "disabled": false
    }
  }
}
```

> パスの区切りは `\\`（バックスラッシュ 2 つ）で記述します。drawio は npx 経由のためパス変更は不要です。

**✔ 確認**: `.kiro/settings/mcp.json` が存在し、terraform の `command` が自分の配置パスになっている。

## Step 5. MCP サーバーの接続確認

1. Kiro で本リポジトリを開く（初回はワークスペースの信頼確認を許可）
2. 左側の Kiro アイコンからサイドパネルを開き、「**MCP Servers**」セクションを確認
3. `terraform Connected` / `drawio Connected` と表示されていれば OK

> mcp.json を変更した場合、Kiro の再起動なしにサイドパネルから反映できます。サーバーが表示されない場合は Kiro を再起動してください。

**✔ 確認**: MCP Servers に terraform / drawio が **Connected** で表示される。

## Step 6. スキルの動作確認

チャット入力欄で `/` を入力し、以下がスラッシュコマンド候補に表示されることを確認します。

- `/generate-design` / `/generate-code` / `/generate-diagram`

**✔ 確認**: 3 つのスキルが候補に表示される。

これでセットアップ完了です 🎉 実際の使い方は **利用手順書** に進んでください。

## セットアップ時のトラブルシューティング

| 症状 | 対処 |
|---|---|
| `eng-repos/` が空 | `git submodule update --init --recursive` を実行（Step 2） |
| Submodule 取得が認証エラー | 対象 CSP プロジェクトのアクセス権・PAT を確認 |
| MCP Servers に terraform / drawio が出ない | `.kiro/settings/mcp.json` の存在と `command` のパスを確認（Step 4）→ Kiro を再起動 |
| terraform MCP が接続エラー | バイナリ単体で起動確認（Step 3）。パスの `\\` 区切りを確認 |
| `/generate-*` がスラッシュ候補に出ない | SKILL.md 保存直後はインデックス遅延あり。少し待つか Kiro を再起動 |
| `/generate-diagram` が失敗する | `node -v`（v18 以上）/ `npx -v` を確認（§0）。台帳に従い Node.js を導入 |
