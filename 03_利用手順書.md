# 利用手順書（開発の流れ）

本書は Kiro を用いて実際に **要件定義 → 基本設計書 → Terraform コード → 構成図** を生成する際の手引きです。初期セットアップが未完了の場合は、先に **導入手順書** を実施してください。

> **ドキュメント体系**: ① 環境仕様書（構成と設計思想）→ ② 導入手順書（初期セットアップ）→ ③ **利用手順書（本書）** → ④ 運用手順書（メンテナンス手順）

---

## 1. 開発フローの原則（最初に必ず読んでください）

本リポジトリの生成フローは、**人間が要所で判断・修正依頼を行い、Kiro が仕様・実装・検証を補助する対話フロー**です。利用にあたって、以下を常に意識してください。

1. **AI の出力を鵜呑みにしない。** 生成された仕様書・設計書・コードは**必ず人の目でレビュー**します。生成物の品質責任は、レビュー・承認した人間にあります。
2. **フェーズごとに人間の承認を挟む。** 各フェーズは承認後に次へ進みます。設計書 → コードは連続実行せず、Kiro が承認なしに次フェーズを自動起動することもありません。
3. **`[要確認]` を放置しない。** インプットから確定できない値は `[要確認]` とマークされます。生成後に必ず洗い出し、解消してから次フェーズへ進みます。
4. **修正はピンポイントで指示する。** 「××の箇所を〇〇に修正して」と依頼し、該当箇所のみ直させます。全体の作り直しは差分レビューを困難にします。
5. **インプットの質が成果物の質を決める。** QA 票・基本設計書・構成図は最新・確定版を配置してください。

**💡 Tips:**

- **夕方頃は Kiro に投げてからレスポンスが遅くなることがあります**（原因はアクセス集中と思われます）。急ぎの生成は時間帯をずらすことも検討してください。
- 生成結果が不安定な場合は、まずモデルがチーム標準（§2）になっているか確認してください。

## 2. 使用モデル

- 本チームで使用できるモデルは **Claude シリーズのみ**です（AWS 側の管理設定で制限済み）。
- チーム標準（デフォルト設定）は **Claude Opus 4.7** です。本リポジトリの生成フローは Opus 4.7 で検証済みであり、**動作保証はこのモデルに対して**行っています。
- 各自、Kiro のモデルセレクターから **Claude シリーズ内で変更可能**ですが、標準以外のモデルでの生成品質・挙動は保証対象外です。生成結果が不安定な場合は、まずモデルを Opus 4.7 に戻して再実行してください。
- 新しいモデルの検証・標準の変更は管理側で定期的に実施します（04 運用手順書参照）。

## 3. スキルの起動方法

### 3-1. 入力ファイルの配置

案件の入力ファイルを **`inputs/<案件フォルダ名>/`** に配置します（チャット添付は不要。各スキルは起動時にこのフォルダをディスク参照します）。ファイル名は正式名のままで構いません — 以下のワイルドカードパターンで判別されます（正: `naming-conventions.md`。`.xlsx` の読み取り方式は `excel-reading.md`）。

```
inputs/{project}/
├── *QA*.xlsx          # QA 票（要件・制約・前提条件）
├── *基本設計書*.xlsx   # 基本設計書（フォーマット・確定値・インフラ構成シート）
└── *.png              # 既存構成図（任意）
```

### 3-2. スキルの起動

本リポジトリを Kiro で開き、スキルを**スラッシュコマンドで明示起動**します。引数には `<案件フォルダ名>`（= `inputs/{project}/` および出力先 `outputs/{cloud}/{project}/` の `{project}`）を渡します。

| やりたいこと | 起動コマンド | 出力先（`outputs/{cloud}/{project}/` 配下） |
|---|---|---|
| 仕様書・設計書を作成（2 フェーズ） | `/generate-design <案件フォルダ名>` | `docs/requirements.md` → `docs/design.md` ＋ **設計書 Excel（入力の基本設計書と同名の .xlsx**・書式を保持したまま書き戻し**）** |
| Terraform コードを作成 | `/generate-code <案件フォルダ名>` | `code/{env}/` |
| 構成図を作成 | `/generate-diagram <案件フォルダ名>` | `docs/architecture.drawio` |

> - `{cloud}` は入力ファイルから判定し、不明な場合は Kiro が確認します。`{env}` は基本設計書の**インフラ構成シート名**（「2-1」等）から読み取ります。
> - 命名・入出力先は `.kiro/steering/naming-conventions.md` を正とします。

**起動例:**

```
（inputs/cms-kiro-test/ に QA 票・基本設計書・構成図を配置してから）
/generate-design cms-kiro-test
```

## 4. 開発フロー（5 ステップ）

1. QA 票・基本設計書・構成図を `inputs/<案件フォルダ名>/` に配置し、`/generate-design <案件フォルダ名>` を実行 → Kiro が **仕様書（requirements.md）** を作成して停止。
2. 仕様書をレビューし、問題なければ「続けて設計書を作成して」と依頼 → Kiro が **基本設計書（design.md ＋ 同名 Excel）** を作成して停止。
3. 設計書をレビューし、修正が必要なら「設計書の××の箇所を〇〇に修正して」と依頼 → Kiro が該当箇所のみ修正（md と Excel の両方に反映）。
4. 設計書に問題が無いことを確認し、`/generate-code <案件フォルダ名>` を実行 → Kiro が **Terraform コード（IaC）** を作成して停止。
5. コードレビュー・デプロイテスト後、修正が必要なら「コードの××の箇所を〇〇に修正して」と依頼 → Kiro が該当箇所のみ修正。

構成図が必要な場合は、設計書確定後に `/generate-diagram <案件フォルダ名>` を実行します。

## 5. 補足（つまずきやすいポイント）

- **設計書 → コードは連続実行しません。** 各フェーズで人間の承認を挟み、Kiro が承認なしに次フェーズや `/generate-code` を自動起動することはありません。
- **インプットはディスク参照です。** 入力は `inputs/{project}/` を参照するため、チャットへの添付・再添付は不要です。別セッションで再開する場合も配置済みのファイルがそのまま参照されます（`design.md` もディスク保存済みのため常に参照可能）。**入力ファイルを差し替えた場合は、該当フェーズを再実行**してください。
- **不明値は `[要確認]`** とマークされます。生成後に必ず洗い出して解消してください。

## 6. コード生成後の検証（レビュー依頼前に実施）

コード生成後は、レビュー依頼の前に以下の後処理・検証を実施します。

**Agent Hooks（generate-code-terraform-checks）導入済みの場合**: Kiro サイドパネルの Agent Hooks から実行します（`.gitignore` 配置 → terraform-docs による成果物 README 生成 → `terraform fmt` → 各環境で `terraform init -backend=false` / `validate` を一括実行し、✅/❌ のサマリを表示します）。❌ がある場合は該当箇所を修正して再実行してください。

**Hook 未導入の場合（手動実行）**:

```bash
terraform fmt -recursive     # 対象: outputs/{cloud}/{project}/code/
terraform init -backend=false && terraform validate   # 各環境ルート（code/{env}/）で実行
```
