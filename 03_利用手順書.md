# 利用手順書（開発の流れ）

本書は Kiro を用いて実際に **要件定義 → 基本設計書 → Terraform コード → 構成図** を生成する際の手引きです。初期セットアップが未完了の場合は、先に **導入手順書** を実施してください。

> **ドキュメント体系**: ① 環境仕様書（構成と設計思想）→ ② 導入手順書（初期セットアップ）→ ③ **利用手順書（本書）** → ④ 運用手順書（メンテナンス手順）

---

## 1. 開発フローの原則（最初に必ず読んでください）

本リポジトリの生成フローは、**人間が要所で判断・修正依頼を行い、Kiro が仕様・実装・検証を補助する対話フロー**です。利用にあたって、以下を常に意識してください。

1. **AI の出力を鵜呑みにしない。** 生成された仕様書・設計書・コードは**必ず人の目でレビュー**します。生成物の品質責任は、レビュー・承認した人間にあります。
2. **フェーズごとに人間の承認を挟む。** 各フェーズは承認後に次へ進みます。設計書 → コードは連続実行せず、Kiro が承認なしに次フェーズを自動起動することもありません。
3. **`[要確認]` を放置しない。** インプットから確定できない値は `[要確認]` とマークされます。生成後に必ず洗い出し、解消してから次フェーズへ進みます。
4. **修正はピンポイントで指示する。** 「××の箇所を〇〇に修正して」と依頼し、該当箇所のみ直させます。全体の作り直しは差分レビューを困難にします。
5. **インプットの質が成果物の質を決める。** QA 票・基本設計書・構成図は最新・確定版を添付してください。

**💡 Tips:**

- **夕方頃は Kiro に投げてからレスポンスが遅くなることがあります**（原因はアクセス集中と思われます）。急ぎの生成は時間帯をずらすことも検討してください。
- 設計レビュー・修正のやり取りが長く続いた後や別セッションで再開する場合は、インプットを再添付してください（§4 補足参照）。

## 2. スキルの起動方法

本リポジトリを Kiro で開き、スキルを**スラッシュコマンドで明示起動**します。起動時は `<案件フォルダ名>`（= 出力先 `Outputs/{cloud}/{project}/` の `{project}`）を引数で渡し、インプット（QA 票・基本設計書・構成図）を**チャット添付**または `#File` で指定します。

| やりたいこと | 起動コマンド | 出力先 |
|---|---|---|
| 仕様書・設計書を作成（2 フェーズ） | `/generate-design <案件フォルダ名>` | `docs/requirements.md` → `docs/design.md` |
| Terraform コードを作成 | `/generate-code <案件フォルダ名>` | `code/{env}/` |
| 構成図を作成 | `/generate-diagram <案件フォルダ名>` | `docs/architecture.drawio` |

> 出力先はすべて `Outputs/{cloud}/{project}/` 配下です。`{cloud}` は添付インプットから判定し、不明な場合は Kiro が確認します。命名・出力先は `.kiro/steering/naming-conventions.md` を正とします。

**起動例:**

```
（QA 票・基本設計書・構成図をそれぞれ添付してから）
/generate-design cms-aws-kiro-test
```

## 3. 開発フロー（5 ステップ）

1. QA 票・基本設計書・構成図を添付し、`/generate-design <案件フォルダ名>` を実行 → Kiro が **仕様書（requirements.md）** を作成して停止。
2. 仕様書をレビューし、問題なければ「続けて設計書を作成して」と依頼 → Kiro が **基本設計書（design.md）** を作成して停止。
3. 設計書をレビューし、修正が必要なら「設計書の××の箇所を〇〇に修正して」と依頼 → Kiro が該当箇所のみ修正。
4. 設計書に問題が無いことを確認し、`/generate-code <案件フォルダ名>` を実行 → Kiro が **Terraform コード（IaC）** を作成して停止。
5. コードレビュー・デプロイテスト後、修正が必要なら「コードの××の箇所を〇〇に修正して」と依頼 → Kiro が該当箇所のみ修正。

構成図が必要な場合は、設計書確定後に `/generate-diagram <案件フォルダ名>` を実行します。

## 4. 補足（つまずきやすいポイント）

- **設計書 → コードは連続実行しません。** 各フェーズで人間の承認を挟み、Kiro が承認なしに次フェーズや `/generate-code` を自動起動することはありません。
- **インプットの再添付**: 同一セッションで `design.md` 承認直後に `/generate-code` を起動する場合、前フェーズで添付した QA 票・基本設計書は会話に残っているため再添付は不要です。**別セッションでの起動、または設計レビュー・修正が長く続いた後は再添付**してください（`design.md` はディスク保存済みのため常に参照可能）。
- **不明値は `[要確認]`** とマークされます。生成後に必ず洗い出して解消してください。

## 5. コード生成後の検証（レビュー依頼前に実施）

コード生成後は、レビュー依頼の前に以下の後処理・検証を実施します。

**Agent Hooks（post-generate-code）導入済みの場合**: Kiro サイドパネルの Agent Hooks から `post-generate-code` を実行します（terraform-docs による成果物 README 生成 → `terraform fmt` → 各環境で `terraform init -backend=false` / `validate` を一括実行し、✅/❌ のサマリを表示します）。❌ がある場合は該当箇所を修正して再実行してください。

**Hook 未導入の場合（手動実行）**:

```bash
terraform fmt -recursive     # 対象: Outputs/{cloud}/{project}/code/
terraform init -backend=false && terraform validate   # 各環境ルート（stg/ prd/）で実行
```
