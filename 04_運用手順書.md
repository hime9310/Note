# 運用手順書（メンテナンス手順）

本書は Kiro に対する **チューニング（ルール・スキル・設定の変更）** と、Submodule・MCP の保守を行う際の手引きです。変更シナリオごとに「修正するファイル」と「注意事項」を記載します。

> **ドキュメント体系**: ① 環境仕様書（構成と設計思想）→ ② 導入手順書（初期セットアップ）→ ③ 利用手順書（開発の流れ）→ ④ **運用手順書（本書）**

---

## 0. メンテナンスの前提（必ず読んでください）

本リポジトリのディレクトリ構成は、**環境仕様書 §4「構成の設計思想」**（steering と skills の分離／段階的読み込み／正の一元化／明示起動／共有物と個人・案件固有物の分離）に基づいて構成されています。

- **ファイルの更新・追加の際は、この思想に乗っ取ってください。** 迷ったら環境仕様書 §4 の「配置の判断基準」に立ち返ります。
- **もし思想に反する更新が発生する場合は、環境仕様書の記載も併せて変更してください。** 実態と仕様書が乖離した状態が最も危険です（変更レビュー時に環境仕様書の更新有無も確認します）。

**共通原則:**

- 「**正（唯一の定義場所）**」だけを直す。同じ情報を複数ファイルに書かない（転記は同期漏れの原因）。
- ファイルを改名・統廃合したら `.kiro/` 全体・`README.md`・`AGENTS.md`・本 4 分冊を grep して参照切れを潰す。
- ドキュメントのみの変更はコミットプレフィックス `docs:` を使用する。

## 1. Terraform コーディング規約（ファイル命名・バージョン・backend 等）を変更する

**修正するファイル**: `.kiro/steering/coding-conventions.md`

**注意事項:**

- fileMatch（`**/*.tf`）のため、`.tf` 作業時に自動適用されます。他ファイルへの転記は不要です。
- 本ファイルは「公式ベストプラクティスへの委譲 + チーム差分のみ記載」の構成です。一般的な作法を追記せず、**チーム固有の決定事項だけ**を書いてください。
- 公式と意図的に競合するルール（例: リソースブロック名 `this`）を消す・変える場合は、生成済みコードとの整合に注意してください。

## 2. 命名規約（案件フォルダ名・入出力先等）を変更する

**修正するファイル**: `.kiro/steering/naming-conventions.md`

**注意事項:**

- always 読み込みのため保存後すぐ全対話に反映されます。分量を増やしすぎるとクレジット・文脈を圧迫するので簡潔に保ってください。
- 入力ファイルのワイルドカードパターン（`*QA*.xlsx` / `*基本設計書*.xlsx` / `*.png`）を変える場合、各スキルのファイル判別に直結します。既存案件の入力ファイル名で判別できなくなるものがないか確認してください。
- 既存案件の `outputs/` フォルダ名は自動では変わりません。命名変更は新規案件から適用する運用としてください。
- 出力先の変更は利用手順書 §3 の表にも影響します。併せて更新してください。

## 3. DCS 環境定義・CMS 標準動作を変更する

**修正するファイル**: `.kiro/steering/dcs-env-definitions.md`

**注意事項:**

- inclusion は manual です。読み忘れ防止は generate-design / generate-code の SKILL.md 冒頭の「必読」指示で担保しているため、**指示文が消えていないか併せて確認**してください。
- CMS 標準動作（SG/NSG・バックアップ・暗号化・払い出し範囲）は設計書・コード双方に影響します。変更後は両スキルで 1 回ずつ試験生成して反映を確認してください。

## 4. 生成手順・出力先パスを変更する

**修正するファイル**: 該当する `.kiro/skills/generate-*/SKILL.md`

**注意事項:**

- 出力先は SKILL.md 内の**パス直指定が正**です。利用手順書・AGENTS.md に記載している出力先とズレないよう、変更時は両ドキュメントも grep して更新してください。
- 「引数（案件フォルダ名）チェック」「dcs-env-definitions 必読」「人間承認で停止」のセクションは運用の要です。手順を書き換える際も削除しないでください。
- **設計書 Excel の書き戻し手順**（元ファイルのコピー・アンカー駆動での上書き・行操作時の結合セル/スタイル再適用等）は generate-design の SKILL.md が正です。`excel-reading.md` 側には重複定義しないでください。

## 5. CSP 別の記入ルール・リソース略称を変更する

**修正するファイル**: 該当する `.kiro/skills/*/parts/{cloud}.md`（aws / azure / gcp）

**注意事項:**

- SKILL.md 側には書かないでください（重複禁止）。SKILL.md は「案件 CSP の parts を 1 つだけ読む」構造を前提にしています（環境仕様書 §4 思想②）。
- 見出し ID（`AWS-x` / `Azure-x` / `GCP-x`）の規約を維持してください。

## 6. 設計書フォーマット / モジュールカタログを変更する

**修正するファイル:**

- 設計書フォーマット → `generate-design/references/{cloud}-doc-format.md`
- モジュールカタログ → `generate-code/references/{cloud}-module-catalog.md`

**注意事項:**

- モジュールの inputs/outputs の正は eng-repos 各モジュールの `variables.tf` / `outputs.tf` です。カタログには要約と参照パスのみを書き、詳細を転記しないでください（Submodule 更新時に乖離するため）。

## 7. Excel 入出力ルールを変更する

**修正するファイル**: `.kiro/steering/excel-reading.md`（読み取り範囲・使用ライブラリの方針）／書き戻し手順の変更は generate-design の `SKILL.md`（そちらが正）

**注意事項:**

- always 読み込みのため簡潔に保ってください。
- **読み取り範囲表は基本設計書テンプレートのシート命名と連動**しています（「2-」+数字で始まり「インフラ構成」を含むシートのみ読む等）。テンプレート改訂時は本表も併せて見直してください。`{env}` の読み取り元（インフラ構成シート名）にも影響します。
- **書き込みは openpyxl のセル API を使用**し、`pandas.DataFrame.to_excel` は使用しない原則を崩さないでください（書式・結合セル・シート構成を破壊するため）。
- Python の追加モジュールインストールは行わない方針です（`pip install` を許可するのは openpyxl / pandas のみ。追加が必要な場合は導入手順書 §0 の必須ツール表・IaC 利用ソフトウェア管理台帳の更新とセットでチーム合意を取ってください）。

## 8. MCP の接続設定を変更する

**修正するファイル**: `.kiro/settings/mcp.json`（各自ローカル）

**注意事項:**

- `mcp.json` は Git 管理外です。**チームで共有すべき変更（サーバ追加・autoApprove 変更等）は `mcp.example.json` にも反映**し、チームに周知してください。
- mcp.json の変更は Kiro 再起動なしにサイドパネルから反映できます。反映されない場合のみ再起動してください。

## 9. .kiro 配下のファイルを改名・追加・削除する

**修正するファイル**: 対象ファイル + 参照している全ファイル

**注意事項:**

- 改名・削除時は `.kiro/` 全体・`README.md`・`AGENTS.md`・本 4 分冊を **grep して参照切れを潰す**こと（steering 間・SKILL からの相対リンクが多いため）。
- 新しいスキルを追加する場合、命名は英語「動詞+目的語」（例: `generate-xxx`）で統一し、出力先パス・必読 steering・共通ルールを SKILL.md 内で完結させてください（環境仕様書 §4 思想①②）。
- manual の steering を追加した場合は、**それを必要とする各 SKILL.md に「必ず読み込む」指示を追記**してください（追記漏れ＝読み忘れになります）。
- 配置に迷った場合は環境仕様書 §4 の判断基準に従い、思想と合わない場合は環境仕様書側の更新も検討してください（§0）。

## 10. Submodule（eng-repos）を更新・追加する

**修正するファイル**: `eng-repos/`（Submodule ポインタ）、追加時は `.gitmodules`

| 操作 | コマンド |
|---|---|
| pull 後に最新取得 | `git pull` → `git submodule update --init --recursive` |
| リモート最新へ更新 | `git submodule update --remote --recursive` |
| 特定 Submodule だけ更新 | `git submodule update --remote eng-repos/AWS/<repo-name>` |
| 新規追加 | `git submodule add <repository-url> eng-repos/<CSP>/<repo-name>` |

**注意事項:**

- 更新後は**必ずメインリポジトリ側でポインタ変更をコミット・プッシュ**してチームへ共有してください（手元だけ更新しても他メンバーには反映されません）。

```bash
git add eng-repos
git commit -m "modify: eng-repos サブモジュールを最新へ更新"
```

- `eng-repos/` 配下のファイルを本リポジトリで**直接編集・コミットしない**でください。変更は各 Submodule の元リポジトリで行います。
- Submodule は detached HEAD でチェックアウトされます。元リポジトリで作業する場合は明示的にブランチを切り替えてください。

## 11. MCP サーバーをバージョン更新する

**修正するファイル**: なし（バイナリの差し替えのみ）

**Terraform MCP Server:**

1. https://releases.hashicorp.com/terraform-mcp-server/ で最新バージョンを確認
2. 新バイナリをダウンロード・解凍
3. 既存バイナリを上書き（パスは同じままで OK）
4. Kiro を再起動（`mcp.json` の変更は不要）

**draw.io MCP Server:** npx 経由のため自動的に最新版が使用されます。手動更新は不要です。

## 12. Kiro 本体をバージョン更新する

**修正するファイル**: なし（アプリの再インストール）＋ 02 導入手順書の記載確認

**注意事項:**

- **1.0 リリース以降、自動アップデートは停止中**（再開時期未定）です。更新は https://kiro.dev/ で新バージョンを確認し、インストーラを再取得して適用します。
- チームでバージョンを揃えるため、更新は個人判断で行わず、チーム内で確認のうえ一斉に実施してください。
- 更新後は MCP 接続（terraform / drawio）とスラッシュコマンド表示を再確認してください（02 導入手順書 Step 5〜6）。
- 自動アップデートが再開された場合は、本節と 02 導入手順書 Step 1 の記載を更新してください。

## 13. 使用可能モデルの方針を変更する・新モデルを検証する

**修正する場所**: AWS マネジメントコンソール（IAM Identity Center 配下の Kiro 管理設定）＋ 03 利用手順書 §2・02 導入手順書 Step 1

**注意事項:**

- 使用可能モデルは AWS コンソール側で **Claude シリーズのみ**に制限しています。許可モデルの追加・変更はコンソールで実施します。
- **新しいモデルがリリースされた場合は定期的にチェック**し、採用する場合は次の手順で切り替えます:
  ① 検証用案件で `/generate-design` `/generate-code` `/generate-diagram` を一巡実行して品質・挙動を確認 → ② 問題なければデフォルトモデルを変更 → ③ 03 利用手順書 §2 の「チーム標準」「動作保証」の記載と 02 導入手順書 Step 1 の確認項目を更新 → ④ チームへ周知。
- 検証を経ていないモデルをデフォルトにしないでください（動作保証の記載と実態が乖離します）。

## 14. トラブルシューティング（運用時）

| 症状 | 対処 |
|---|---|
| `eng-repos/` が空 | `git submodule update --init --recursive` を実行 |
| Submodule 取得が認証エラー | 対象 CSP プロジェクトのアクセス権・PAT を確認 |
| Submodule が古い | `git submodule update --remote --recursive`（更新後はポインタをコミット） |
| `inputs/` の中身・`outputs/` がコミットされない | 仕様です（`.gitignore` で除外。`inputs/` はフォルダのみ `.gitkeep` で維持）。共有が必要な成果物は別リポジトリで管理 |
| スキルが入力ファイルを見つけられない | `inputs/<案件フォルダ名>/` の配置とファイル名のワイルドカード（`*QA*.xlsx` / `*基本設計書*.xlsx`）を確認 |
| Excel の読み取り・書き込みが失敗する | `python -c "import openpyxl, pandas"` を確認（02 導入手順書 §0）。基本設計書のシート名が読み取り範囲（「2-」+数字・インフラ構成）に合っているか確認 |
| `/generate-*` がスラッシュ候補に出ない | SKILL.md 保存直後はインデックス遅延あり。少し待つか Kiro を再起動 |
| MCP Servers に terraform / drawio が出ない | 導入手順書 Step 4〜5 を確認（mcp.json の存在・パス）→ Kiro を再起動 |
| `/generate-diagram` が失敗する | 導入手順書 §0 を確認（`node -v` v18 以上 / `npx -v`） |
| terraform MCP が接続エラー | 導入手順書 Step 3 を確認（バイナリ単体で起動確認・`\\` 区切り） |
| `.tf` 生成が MCP エラーで止まる | 規約上、MCP 未確認のままの続行は禁止。接続復旧後に再実行 |
