# 基本設計書 生成手順

Kiroが基本設計書を作成する際の参照先・手順・ルールを定義します。

---

## 1. 参照すること

以下を必ず参照すること（チャットで指定がなくても自動で読み込む）。

### AWS / GCP の場合

| 優先度 | ファイル | 目的 |
|---|---|---|
| 必須 | `inputs/{cloud}/{project}/*基本設計書*.xlsx` | シート構成・項目名・フォーマットの踏襲 |
| 必須 | `inputs/{cloud}/{project}/*QA*.xlsx` | 要件・制約・前提条件の反映 |

> AWSとGCPの場合、パラメーターシートはインフラ基本設計書.xlsx 内に含まれることが多い。
> その場合は別途パラメーターシートの参照は不要。

### Azure の場合

| 優先度 | ファイル | 目的 |
|---|---|---|
| 必須 | `inputs/azure/{project}/*基本設計書*.xlsx` | シート構成・項目名・フォーマットの踏襲 |
| 必須 | `inputs/azure/{project}/*QA*.xlsx` | 要件・制約・前提条件の反映 |
| 必須 | `inputs/azure/{project}/*環境定義書*.xlsx` | パラメーター値の埋め込み（別ファイル） |

---

## 2. 出力形式

- **Markdown (.md)** を基本とする
- Excelのシート構成 → Markdownの章立てに対応させること
- 表形式のデータはMarkdownテーブルで表現すること
- 出力先：`specs/{cloud}/{project}/design.md`

---

## 3. 作成ルール

- Excelの項目構成・章立てを変えないこと
- 不明・未確定の値は `[要確認]` とマークすること（空白禁止）
- 構成図（*.png / *.drawio）が inputs 配下にある場合は内容を参照し、
  アーキテクチャの記述に反映すること
- 改善余地がある場合は該当箇所に `[提案]` としてコメントを追記すること
- Kiroは人間の承認なしに次のPhaseへ進まないこと
