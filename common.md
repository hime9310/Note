# 共通ルール(Design Doc ワークスペース)

このワークスペースは社内向けの設計ドキュメント成果物(PowerPoint・Excel 等)を Kiro で生成するためのもの。以下は全フェーズ・全スキルに常時適用する共通ルール。各スキルの詳細手順はここで再定義せず、該当 SKILL に従う。

## スキルの役割分担
- PowerPoint 資料の生成は `pptx-builder` スキルに従う。
- Excel 帳票の生成は `excel-builder` スキルに従う。
- 各スキルの入口は `SKILL.md`。`reference/` 配下のファイルは SKILL.md が指示したときのみ読み込み、常時は読み込まない。
- 共通事項は本 steering を正とする。SKILL 側は同じ内容を再記述せず、これを参照する。

## 実行環境の前提
- コマンド・スクリプトはワークスペース直下で実行する。相対パスはワークスペースルートから解決する。
- 生成は python3 で行う(PowerPoint は python-pptx、Excel は openpyxl 等)。
- `reference/` 配下のテンプレート・参照元ファイル(.potx 等)は読み取り専用として扱い、上書き保存しない。成果物は必ず別名で保存する。

## 成果物・作業ディレクトリ
- 成果物は `Outputs/{要件名}/` 配下に保存する。ディレクトリが無ければ作成する。
- `{要件名}` は作業中の spec フォルダ名に一致させる。spec が `{cloud}/{project}` 構造ならそれを踏襲する(例:`Outputs/aws/case-001/`)。spec を使わない単発生成では、ユーザーが指定した要件名を用いる。
- ディレクトリ作成は保存直前にコードで行う(`os.makedirs(path, exist_ok=True)`)。冪等に保ち、絶対パスをハードコードしない。
- `Outputs/` と `specs/` は案件固有の成果物であり Git 管理外(.gitignore 対象)。共通基盤リポジトリには含めない。

## クレジット消費のガードレール
- 決まりきった処理(ファイルコピー・ディレクトリ作成・整形など)は LLM に行わせず、コードまたはシェルコマンドで決定的に処理する。
- 目次(TOC)を steering・SKILL に書かない。参照はマークダウンリンクではなく相対パスの直接指定とし、更新時のリンク崩れとトークン増を避ける。
- 生成→検証の自己修正ループは最大3回まで。上限に達したらエラー内容を報告して停止し、無限リトライしない。
- ファイルの全文読み込みや大量の標準出力を避ける。シェル実行時の出力は1行サマリ程度に抑える(stdout はエージェントのコンテキストに取り込まれるため)。

## 全成果物に共通の遵守事項
- 著作権:他者の写真・他社ロゴ等を無断利用しない。
- 情報区分:機密区分ラベル(Confidential 等)はテンプレートの位置・体裁を維持する。
- 色覚多様性に配慮し、色だけで情報を区別しない。
- 詳細なブランド要件(フォント・カラー・レイアウト・免責ページ等)は各 SKILL の定義に従う。
