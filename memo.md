

**Kiroに貼るプロンプト(HOOK説明パワポ・表紙→目次→サマリ・全11枚)**

> pptx-builder スキルを使って、社内チーム向けの説明用パワポを作成してください。用途は配布/ハンドアウトなので白背景。SKILLのルール(タイトル24pt、本文はワーキングエリア内、コピーライト右下、末尾に内部向け免責ページ)に従い、`Outputs/{要件名}/` に別名で保存してください。テーマは「Kiro Agent Hooks の活用」。全11枚、構成は以下。
>
> 1. 表紙：タイトル「Kiro Agent Hooks の活用提案 — モジュール配置の自動化とクレジット最適化」
> 2. 目次：§5の対応表にある目次レイアウト(白)を使う。項目は「1. Hookとは / 2. 現状の課題 / 3. 解決策 / 4. 導入イメージ / 5. 注意点 / 6. 次アクション」
> 3. サマリ(エグゼクティブサマリ)：モジュールコピーをsteering(LLM)からHook(シェル)へ移すことで、生成のたびのトークン消費をほぼゼロにできる、という結論を先に述べる
> 4. Hookとは：IDEのイベント(ファイル作成/保存/削除・手動)で発火する自動トリガー。アクションは「エージェントプロンプト(Ask Kiro)」と「シェルコマンド(Run Command)」の2種類
> 5. 現状の課題：参照リポジトリからのモジュール一式コピーをsteering(=LLM)にやらせており、コード生成のたびにトークン=クレジットを毎回消費している
> 6. 解決策：コピーは決定的処理。Run Command(シェル)型Hookに移せばLLMを介さずOSのファイル操作で完結し、消費はほぼゼロ。steeringからは該当ロジックを削減
> 7. 導入イメージ：トリガーは手動/ファイル作成、アクションは冪等なコピースクリプト。HookのJSONは `.kiro/hooks/` にコミットしてチーム共有。Before(steeringでLLMがコピー)／After(Hookがシェルで配置)の対比を図示
> 8. 注意点：①必ずRun Command(Ask Kiroだと削減にならない) ②stdoutはコンテキストに入るので大量出力しない ③冪等にして編集済みを壊さない ④Pre系トリガーは軽く＋timeout ⑤steeringは整理しつつ「ベースは配置済み、ゼロから作るな」の一文だけ残す
> 9. 次アクション：PoCで1本作成 → クレジット差分を計測 → 横展開
