KiroのAgent Hooksは、IDE内のイベント(ファイル保存・作成、ツール実行前後など)をトリガーに、**シェルコマンド**または**エージェントプロンプト**を自動実行する仕組みです。フックは `.kiro/hooks/`(ワークスペース単位)または `~/.kiro/hooks/`(ユーザー単位)にJSONファイルとして保存され、自然言語で依頼・フォーム入力・JSON直書きの3通りで作成できます。 

## クレジット最適化の要点

まず一番重要なポイントです。アクションには2種類あり、コスト特性が全く違います。

- **`command`(シェルコマンド)**: エージェントを起動しない → **クレジット消費ゼロ**
- **`agent`(エージェントプロンプト)**: 別セッションでエージェントが起動 → クレジット消費あり

つまり「fmt実行」「terraform-docs実行」のような**決定的な処理はcommandフック化**するのが最適化の本丸です。steeringに書いてあると毎ターンのコンテキストトークンを食い、かつエージェントが「実行を判断する」ためのやり取りも発生しますが、commandフックならどちらもゼロになります。

## 実装例

`.kiro/hooks/terraform.json` など1ファイルにまとめられます:

```json
{
  "version": "v1",
  "hooks": [
    {
      "name": "tf-fmt-on-save",
      "trigger": "PostFileSave",
      "matcher": "\\.tf$",
      "action": {
        "type": "command",
        "command": "terraform fmt \"$KIRO_FILE_PATH\""
      },
      "timeout": 30
    },
    {
      "name": "tf-fmt-on-create",
      "trigger": "PostFileCreate",
      "matcher": "\\.tf$",
      "action": {
        "type": "command",
        "command": "terraform fmt \"$KIRO_FILE_PATH\""
      }
    },
    {
      "name": "tf-docs-readme",
      "trigger": "PostFileSave",
      "matcher": "modules/.*\\.tf$",
      "action": {
        "type": "command",
        "command": "cd $(dirname \"$KIRO_FILE_PATH\") && terraform-docs markdown table . > README.md"
      },
      "timeout": 60
    }
  ]
}
```

トリガーは `PostFileCreate`(エージェントによるファイル作成後)、`PostFileSave`(保存後)、`PreToolUse` / `PostToolUse`(ツール実行前後、matcherはツール名)、`PreTaskExec` / `PostTaskExec`(Specタスク前後)、`UserPromptSubmit`、`SessionStart`、`Stop` などがあり、matcherは正規表現でファイルパスやツール名をフィルタします。 

## モジュールコピー(判断が必要なタスク)の扱い

「どの既存モジュールを参照・コピーするか」はLLMの判断が必要なので、これをフック化するなら `agent` アクションになります:

```json
{
  "name": "module-reuse",
  "trigger": "PostFileCreate",
  "matcher": "envs/.*\\.tf$",
  "action": {
    "type": "agent",
    "prompt": "作成されたtfファイルを分析し、modules/ 配下の既存モジュールで再利用できるものがあればsource参照に置き換えてください。判断基準は .kiro/steering/module-selection.md に従うこと"
  }
}
```

ただしクレジット観点では注意が要ります。**agentフックは発火のたびに新しいエージェントセッションが走る**ので、ファイル作成が多いと本体の会話でsteeringを読ませるより高くつく可能性があります。おすすめの整理は:

1. **fmt / terraform-docs / validate** → commandフック(ゼロコスト、steeringから該当記述を削除)
1. **モジュール選定** → 頻度が低いならSKILL/steeringのまま本体セッションで処理、頻度が高く判断がパターン化できるなら「候補モジュール一覧をスクリプトで列挙するcommandフック + 短いsteering」のハイブリッドが安い
1. `SessionStart` や `UserPromptSubmit` のcommandフックはexit 0 でSTDOUTがコンテキストに追加されるので、「モジュール一覧を  `find modules/ -name README.md` で注入する」ような使い方をすれば、エージェント判断の材料をゼロクレジットで渡せます。

## 運用の小技

- `enabled: false` で削除せずスキップでき、パネルの目アイコンでもトグル可能 
- `PreToolUse` / `UserPromptSubmit` / `PreTaskExec` では exit code 2 で実行をブロックでき、STDERRがエージェントに返るので、「fmt未適用ならタスク完了させない」ようなガードレールも組めます 
- `.kiro/hooks/` をGit管理すればチーム全員に同じ自動化が配布されます

まとめると、御社のケースでは fmt と terraform-docs は即commandフック化で確実にクレジット削減、モジュールコピーだけは「発火頻度 × agentセッション費用」と「steering常時ロード費用」を比べて決めるのが良いと思います。
