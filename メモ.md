KiroのAgent Hooksは、IDE内のイベント(ファイル保存・作成、ツール実行前後など)をトリガーに、**シェルコマンド**または**エージェントプロンプト**を自動実行する仕組みです。フックは `.kiro/hooks/`(ワークスペース単位)または `~/.kiro/hooks/`(ユーザー単位)にJSONファイルとして保存され、自然言語で依頼・フォーム入力・JSON直書きの3通りで作成できます。 

## クレジット最適化の要点

まず一番重要なポイントです。アクションには2種類あり、コスト特性が全く違います。

- **`command`(シェルコマンド)**: エージェントを起動しない → **クレジット消費ゼロ**
- **`agent`(エージェントプロンプト)**: 別セッションでエージェントが起動 → クレジット消費あり

つまり「fmt実行」「terraform-docs実行」のような**決定的な処理はcommandフック化**するのが最適化の本丸です。steeringに書いてあると毎ターンのコンテキストトークンを食い、かつエージェントが「実行を判断する」ためのやり取りも発生しますが、commandフックならどちらもゼロになります。

## 実装例

`.kiro/hooks/terraform.json` など1ファイルにまとめられます:

```json
{
  "version": "v1",
  "hooks": [
    {
      "name": "tf-fmt-on-save",
      "trigger": "PostFileSave",
      "matcher": "\\.tf$",
      "action": {
        "type": "command",
        "command": "terraform fmt \"$KIRO_FILE_PATH\""
      },
      "timeout": 30
    },
    {
      "name": "tf-fmt-on-create",
      "trigger": "PostFileCreate",
      "matcher": "\\.tf$",
      "action": {
        "type": "command",
        "command": "terraform fmt \"$KIRO_FILE_PATH\""
      }
    },
    {
      "name": "tf-docs-readme",
      "trigger": "PostFileSave",
      "matcher": "modules/.*\\.tf$",
      "action": {
        "type": "command",
        "command": "cd $(dirname \"$KIRO_FILE_PATH\") && terraform-docs markdown table . > README.md"
      },
      "timeout": 60
    }
  ]
}
```

トリガーは `PostFileCreate`(エージェントによるファイル作成後)、`PostFileSave`(保存後)、`PreToolUse` / `PostToolUse`(ツール実行前後、matcherはツール名)、`PreTaskExec` / `PostTaskExec`(Specタスク前後)、`UserPromptSubmit`、`SessionStart`、`Stop` などがあり、matcherは正規表現でファイルパスやツール名をフィルタします。 

## モジュールコピー(判断が必要なタスク)の扱い

「どの既存モジュールを参照・コピーするか」はLLMの判断が必要なので、これをフック化するなら `agent` アクションになります:

```json
{
  "name": "module-reuse",
  "trigger": "PostFileCreate",
  "matcher": "envs/.*\\.tf$",
  "action": {
    "type": "agent",
    "prompt": "作成されたtfファイルを分析し、modules/ 配下の既存モジュールで再利用できるものがあればsource参照に置き換えてください。判断基準は .kiro/steering/module-selection.md に従うこと"
  }
}
```

ただしクレジット観点では注意が要ります。**agentフックは発火のたびに新しいエージェントセッションが走る**ので、ファイル作成が多いと本体の会話でsteeringを読ませるより高くつく可能性があります。おすすめの整理は:

1. **fmt / terraform-docs / validate** → commandフック(ゼロコスト、steeringから該当記述を削除)
1. **モジュール選定** → 頻度が低いならSKILL/steeringのまま本体セッションで処理、頻度が高く判断がパターン化できるなら「候補モジュール一覧をスクリプトで列挙するcommandフック + 短いsteering」のハイブリッドが安い
1. `SessionStart` や `UserPromptSubmit` のcommandフックはexit 0 でSTDOUTがコンテキストに追加されるので、「モジュール一覧を  `find modules/ -name README.md` で注入する」ような使い方をすれば、エージェント判断の材料をゼロクレジットで渡せます。

## 運用の小技

- `enabled: false` で削除せずスキップでき、パネルの目アイコンでもトグル可能 
- `PreToolUse` / `UserPromptSubmit` / `PreTaskExec` では exit code 2 で実行をブロックでき、STDERRがエージェントに返るので、「fmt未適用ならタスク完了させない」ようなガードレールも組めます 
- `.kiro/hooks/` をGit管理すればチーム全員に同じ自動化が配布されます

まとめると、御社のケースでは fmt と terraform-docs は即commandフック化で確実にクレジット削減、モジュールコピーだけは「発火頻度 × agentセッション費用」と「steering常時ロード費用」を比べて決めるのが良いと思います。

はい、それはフック化に非常に向いています。「指定の内容」つまり中身が固定なら判断が不要なので、**commandフックでクレジット消費ゼロ**にできます。steeringに「gitignoreを作ること」と書いてエージェントに生成させるのは、毎回同じ内容をLLMに出力させる分トークンの無駄なので、むしろ真っ先にフックへ移すべきタスクです。

実装のコツは、テンプレートファイルをリポジトリに置いてコピーする方式にすることです:

```json
{
  "version": "v1",
  "hooks": [
    {
      "name": "gitignore-bootstrap",
      "trigger": "PostFileCreate",
      "matcher": "\\.(tf|py|ts)$",
      "action": {
        "type": "command",
        "command": "[ -f \"$KIRO_WORKSPACE_ROOT/.gitignore\" ] || cp \"$KIRO_WORKSPACE_ROOT/.kiro/templates/gitignore.tpl\" \"$KIRO_WORKSPACE_ROOT/.gitignore\""
      },
      "timeout": 10
    }
  ]
}
```

ポイントは3つあります。

**冪等にする**: `PostFileCreate` はコードファイルを作るたびに発火するので、`[ -f .gitignore ] ||` のガードで「存在しなければコピー」にしておかないと毎回上書きされます。既存の手動編集を守る意味でも必須です。

**テンプレートはGit管理**: `.kiro/templates/gitignore.tpl` のような場所に実体を置けば、内容変更もプルリクでレビューでき、チーム全員に同じものが配布されます。SKILLの中に「gitignoreにはこれを書く」という長文があるなら、それを丸ごとテンプレファイルに移してSKILLから削除でき、コンテキスト削減にもなります。

**トリガーの代替案**: モノレポでディレクトリごとにgitignoreを置きたい場合は、`$KIRO_FILE_PATH` の親ディレクトリを基準にチェックする形にすればOKです。逆に「プロジェクト種別ごとに内容を変えたい」(Terraformなら`.terraform/`、Pythonなら`__pycache__/`など)程度なら、拡張子で分岐するシェルスクリプトを1本書いてフックから呼べば、これもLLM判断なしで済みます。

内容の判断が本当に必要になるのは「リポジトリ構成を読んで最適なgitignoreを推論する」ようなケースだけで、その場合のみagentアクションを検討すれば良い、という整理です。

【共有】KiroのHook機能を一部導入します

お疲れさまです。Kiroの「Agent Hooks」という機能を使って、Terraform周りの定型作業を自動化する取り組みを始めますので共有します。

■ Hookとは？
「ファイルが保存された」「ファイルが作成された」などのイベントをきっかけに、あらかじめ決めた処理を自動実行してくれるKiroの機能です。設定は .kiro/hooks/ フォルダのJSONファイルで管理され、Gitで共有されるので、リポジトリをpullすれば全員に同じ自動化が適用されます。

■ 今回実装する内容(第1弾)
今回は「シェルコマンドを実行するだけ」のタイプのHookのみを実装します。AIエージェントを起動しないため、クレジット消費はゼロです。

1. tfファイル保存時に terraform fmt を自動実行(フォーマット統一)
2. terraform-docs によるREADMEの自動生成・更新
3. 変数定義のチェック
4. コード作成時に所定の .gitignore を自動追加(既にある場合は上書きしません)

これまでsteeringやSKILLでAIにお願いしていた作業の一部が自動化されるため、コンテキスト(トークン)の節約にもなります。

■ 今回見送るもの
「既存モジュールを参照してコピーする」処理は、AIの判断が必要なタイプ(Agentアクション)になり、実行のたびにクレジットを消費します。現在のsteering運用とどちらが安いか比較できていないため、別途相談の場を設けさせてください。
