自動化の選択肢を正確にお答えしたいので、現行Kiroのhookトリガー種別を確認します。

確認できました。トリガーは現在**10種類**あり、「なるべく自動」を実現できるものが増えています。まず一覧です。

| トリガー | 発火タイミング | 今回の用途への適性 |
|---|---|---|
| `fileEdited` / `fileCreated` / `fileDeleted` | globに一致するファイルの保存/作成/削除 | △ 生成中に多重発火 |
| `userTriggered` | パネルのボタン（現在の方式） | ○ 手動再検証用 |
| `promptSubmit` | ユーザーがメッセージ送信した時 | ✕ タイミングが逆 |
| **`agentStop`** | **エージェントが応答を完了した時** | **◎ 本命** |
| `preToolUse` / `postToolUse` | エージェントのツール実行の前後（write/shell等でフィルタ可・preは非0終了でブロック可能） | 別用途（後述） |
| `preTaskExecution` / `postTaskExecution` | Spec機能のタスク状態変化の前後 | ✕ Spec機能未使用 |

**推奨: `agentStop` + runCommand + スクリプト側の変更検知ガード**です。`agentStop` は「エージェントの実行完了時」に発火するので、`/generate-code` が全ファイルを書き終えて停止した瞬間＝理想のタイミングで走ります。ただし**チャット応答や設計書生成の完了でも毎回発火**するため、スクリプト側に「前回実行以降に `.tf` が変わっていなければ即終了（exit 0・無音）」のガードを入れるのが設計の肝です。これで「.tfを生成したターンの直後だけ実質動く」挙動になります。

**① 新しいhook定義**（既存のManual版は残し、2本目として追加）:

```json
{
  "enabled": true,
  "name": "auto-terraform-checks (on agent stop)",
  "description": "エージェント応答完了時に自動実行。前回以降 .tf に変更が無ければ即終了する。",
  "version": "1",
  "when": { "type": "agentStop" },
  "then": {
    "type": "runCommand",
    "command": "python .kiro/hooks/scripts/run_generate_code_terraform_checks.py --auto",
    "timeout": 0
  }
}
```

**`"timeout": 0` が必須**です。runCommandの既定タイムアウトは60秒で、init/validateを含む本パイプライン（実測36秒＋案件次第）は既定値だと切られる可能性があります。

**② スクリプトへのガード追加**（差分3箇所方式）。定数の下に:

```python
STATE_FILE = REPO_ROOT / ".kiro" / "hooks" / ".tfcheck_state"  # ※.gitignore推奨

def tf_changed_since_last_run(code_dir: Path) -> bool:
    tf_files = list(code_dir.rglob("*.tf"))
    if not tf_files:
        return False
    latest = max(f.stat().st_mtime for f in tf_files)
    try:
        last = float(STATE_FILE.read_text().strip())
    except (FileNotFoundError, ValueError):
        return True
    return latest > last

def save_state(code_dir: Path) -> None:
    tf_files = list(code_dir.rglob("*.tf"))
    if tf_files:
        STATE_FILE.write_text(str(max(f.stat().st_mtime for f in tf_files)))
```

`main()` の引数に `ap.add_argument("--auto", action="store_true")` を足し、`code_dir` 確定直後に:

```python
    if args.auto and not tf_changed_since_last_run(code_dir):
        return 0  # 変更なし: 無音で終了（agentStop の空振り）
```

サマリ出力後（`return` の直前）に `save_state(code_dir)` を1行。これでManual実行時は従来どおり毎回フル実行、`--auto`（agentStop経由）時のみガードが効きます。

**知っておくと良い副作用が1つ**: runCommandは終了コード非0のとき **stderrがエージェントに通知される**仕様です。つまりvalidate NGだとKiroに「hookがエラーを返した」と伝わり、Kiroがそのまま修正に動くことがあります（修正→停止→再びagentStop→再検証、という自己修正ループが自然に成立します）。便利な反面、自動修正をレビューなしで走らせたくない場合は、スクリプトのNG詳細をstdout側に留めておく調整も可能です。まずはこの挙動込みで1案件試してみて、チームの好みに合わせるのが良いと思います。

最後に1点訂正を。以前「MCP参照をシステム的に強制する仕組みはIDEにはない」とお答えしましたが、最新ドキュメントで **`preToolUse` がIDEにも存在し、非0終了でツール実行をブロックできる**ことを確認しました。つまり「`.tf` へのwriteをpreToolUseで捕まえ、MCPエビデンスファイルが無ければ書き込み自体をブロック」という本当の強制ゲートが組めます。前回提案した事後検証（③検証層）より強い選択肢なので、MCP強制を進める際はこちらも検討候補に入れてください。
