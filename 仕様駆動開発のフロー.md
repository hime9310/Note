# Kiro 仕様駆動開発（Spec-Driven Development）フロー整理

## 1. 三つの主要機能の概要

### Spec（仕様）

構造化された方法で機能の設計・実装を進めるための仕組み。要件定義 → 設計 → タスクの3段階で、人間とKiroが反復的にドキュメントを作り上げ、そのドキュメントに基づいてKiroが実装を行う。

- 格納場所: `.kiro/specs/{feature-name}/`
- 生成ファイル: `requirements.md`, `design.md`, `tasks.md`

### Steering（ステアリング）

Kiroとのすべて（または特定条件）のやり取りに追加のコンテキストや指示を注入する仕組み。チームのコーディング規約、プロジェクト固有のルール、ビルド・テスト手順などを定義しておくことで、Kiroの出力品質を安定させる。

- 格納場所: `.kiro/steering/*.md`
- 適用モード:
  - **always**（デフォルト）: すべてのやり取りに常に適用
  - **fileMatch**: 特定ファイルパターンに一致するファイルがコンテキストに含まれるとき適用
  - **manual**: ユーザーがチャットで `#ファイル名` で明示的に指定したとき適用

### Skills（スキル）

Kiroが特定のタスクを実行する際に参照する、再利用可能な手順書・ナレッジ。チャットで `#スキル名` で呼び出して使う。

- 格納場所:
  - ワークスペースレベル: `.kiro/skills/`
  - ユーザーレベル: `~/.kiro/skills/`

## 2. 各機能の Input / Output 整理

### Spec

| フェーズ | 人間の Input | Kiro の Output |
|---|---|---|
| 開始 | 機能のラフなアイデア・要望を自然言語で伝える | Spec種別（Feature/Bugfix）とワークフロー（Requirements-first/Design-first）の選択肢を提示 |
| 要件定義 | Kiroが生成した要件ドラフトをレビューし、修正・承認する | `requirements.md`（ユーザーストーリー、受け入れ基準を含む要件定義書） |
| 設計 | Kiroが生成した設計ドラフトをレビューし、修正・承認する | `design.md`（アーキテクチャ図、コンポーネント設計、データモデル、Correctness Properties） |
| タスク化 | Kiroが生成したタスクリストをレビューし、修正・承認する | `tasks.md`（チェックボックス付きの実装タスクリスト） |
| 実装 | タスク実行を指示する（個別 or 一括） | 実際のコード（タスクに基づいて自動実装） |

ポイント: 各フェーズで人間が「承認」してから次に進む。Kiroが勝手に先に進むことはない。

### Steering

| 項目 | 人間の Input | Kiro の Output |
|---|---|---|
| 定義 | `.kiro/steering/` にMarkdownファイルを作成・編集する | なし（Steeringは人間が定義するもの） |
| 適用 | 自動適用 or ファイル操作時に自動適用 or `#` で手動指定 | Steeringの内容を踏まえた応答・コード生成 |

Steeringファイルの例:
```markdown
---
inclusion: always
---
# コーディング規約
- TypeScriptを使用する
- 関数名はcamelCaseとする
- テストは必ずvitest で書く
```

```markdown
---
inclusion: fileMatch
fileMatchPattern: "**/*.tf"
---
# Terraform規約
- リソース名はsnake_caseとする
- 変数にはdescriptionを必ず付ける
```

### Skills

| 項目 | 人間の Input | Kiro の Output |
|---|---|---|
| 定義 | `.kiro/skills/` にMarkdownファイルを作成・編集する | なし（Skillsは人間が定義するもの） |
| 利用 | チャットで `#スキル名` を指定して呼び出す | スキルの手順に従った実行結果 |

## 3. 一連の開発フロー

```
┌─────────────────────────────────────────────────────────┐
│  事前準備（一度だけ / 必要に応じて更新）                      │
│                                                         │
│  [人間] Steering ファイルを作成                            │
│    └─ .kiro/steering/coding-standards.md                │
│    └─ .kiro/steering/terraform-rules.md                 │
│    └─ ...                                               │
│                                                         │
│  [人間] Skills ファイルを作成                              │
│    └─ .kiro/skills/deploy-procedure.md                  │
│    └─ ...                                               │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Spec 開始                                               │
│                                                         │
│  [人間] 「EC2の起動停止スケジュールを作りたい」              │
│         ↓                                               │
│  [Kiro] Feature or Bugfix? → [人間] 選択                 │
│  [Kiro] Requirements-first or Design-first? → [人間] 選択│
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 1: 要件定義                                       │
│                                                         │
│  [Kiro] requirements.md ドラフト生成                      │
│         ↓                                               │
│  [人間] レビュー → 修正依頼 or 承認                        │
│         ↓                                               │
│  [Kiro] 修正を反映（承認されるまで繰り返し）                │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 2: 設計                                           │
│                                                         │
│  [Kiro] design.md ドラフト生成                            │
│         （Steeringの規約を自動的に考慮）                   │
│         ↓                                               │
│  [人間] レビュー → 修正依頼 or 承認                        │
│         ↓                                               │
│  [Kiro] 修正を反映（承認されるまで繰り返し）                │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 3: タスク化                                       │
│                                                         │
│  [Kiro] tasks.md 生成（チェックボックス付きタスクリスト）    │
│         ↓                                               │
│  [人間] レビュー → 修正依頼 or 承認                        │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  Phase 4: 実装                                           │
│                                                         │
│  [人間] 「タスク実行して」                                 │
│         ↓                                               │
│  [Kiro] tasks.md のタスクを順番に実装                      │
│         （Steering + Spec の内容に基づいてコード生成）      │
│         ↓                                               │
│  [人間] 生成されたコードをレビュー                          │
└─────────────────────────────────────────────────────────┘
```

## 4. 各機能の関係性

```
Steering（常時バックグラウンドで適用）
    │
    │  コーディング規約・プロジェクトルールを注入
    │
    ▼
Spec（開発の主軸フロー）
    │
    │  requirements.md → design.md → tasks.md → 実装
    │
    ▼
Skills（必要に応じて呼び出し）
    │
    │  デプロイ手順、特定の作業手順など
    │
    ▼
成果物（コード、設定ファイル等）
```

- **Steering** は「Kiroの振る舞いを制御するルール」。Specの全フェーズに暗黙的に影響する
- **Spec** は「何を作るかを段階的に定義し、実装まで導くフロー」。開発の主軸
- **Skills** は「特定の作業を実行するための手順書」。Specとは独立して使える

## 5. チーム導入時のポイント

| 観点 | 推奨事項 |
|---|---|
| Steering の管理 | Git管理し、チーム共通の規約として `.kiro/steering/` に配置。PRレビュー対象にする |
| Spec の粒度 | 1つのSpecは1つの機能単位。大きすぎる機能は分割する |
| レビュータイミング | 各フェーズ（要件→設計→タスク）の承認前に人間がしっかりレビューする。ここが品質の要 |
| Skills の活用 | チーム固有のデプロイ手順やトラブルシュート手順をSkillsとして共有する |
| Spec ファイルの参照 | Specドキュメント内で `#[[file:openapi.yaml]]` のように外部ファイルを参照可能。既存のAPI仕様書等を活用できる |
