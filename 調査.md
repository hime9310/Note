- 実行環境: 対象アカウントの CloudShell(リージョン切替はコマンド内の `--region` で対応)

## 背景と本手順の位置づけ

残る立証ポイントは次の3点であり、本手順①〜④で全て埋まる。

| 立証ポイント | 対応する手順 |
|---|---|
| A. 直近(7月)も同一要因=redlockが主因として継続しているか | ①+④ |
| B. redlock が Prisma Cloud のロールである直接証拠 | ② |
| C. 導入日の客観証拠(ログ保持期限に依存しない裏付け) | ③ |

## 共通設定(最初に1回実行)

①と④は「④の行数 ÷ ①の総件数 = redlock比率」を計算するため、必ず同じ期間を使うこと。24時間窓にするのは、1時間だと時間帯による偏りが出る可能性があるのに対し、24時間ならスキャン周期(約1時間毎×24回)を平均化でき、かつ取得時間も許容範囲のため。

```bash
START=2026-07-21T00:00:00Z
END=2026-07-22T00:00:00Z
```

## ① 総イベント件数の確認(TSV出力なし・画面確認のみ)

**何のコマンドか**: CloudTrailイベント履歴(lookup-events)から、対象24時間の全イベント件数を東京(ap-northeast-1)とus-east-1でカウントする。比率計算の「分母」を取るためのコマンド。

```bash
for R in ap-northeast-1 us-east-1; do
  T=$(aws cloudtrail lookup-events --region $R \
      --start-time $START --end-time $END \
      --query 'length(Events)' --output text)
  echo -e "$R\ttotal=$T"
done
```

**結果で何がわかるか**: リージョン別の総イベント件数。出力は2行のみなのでTSV保存は不要、数値をメモしてExcelの調査ブックに転記する。想定は1リージョンあたり1.5〜2万件前後(日次約40万件÷約21リージョンより)。

## ② redlockが使用しているIAMロールARNの特定(1件のみ取得・保存不要)

**何のコマンドか**: redlockのイベントを1件だけ取り出し、生JSONから実行主体のARN(userIdentity.arn)を抽出する。

```bash
aws cloudtrail lookup-events --region ap-northeast-1 \
  --lookup-attributes AttributeKey=Username,AttributeValue=redlock \
  --start-time $START --end-time $END --max-items 1 \
  --query 'Events[0].CloudTrailEvent' --output text \
| python3 -c "import sys,json; e=json.load(sys.stdin); print(e['userIdentity']['arn'])"
```

**結果で何がわかるか**: `arn:aws:sts::＜ID＞:assumed-role/<ロール名>/redlock` という形式のARNが得られる。redlockは「セッション名」であり、このコマンドで「どのIAMロールがそのセッションを発行したか」が判明する。ロール名がPrisma系(PrismaCloud、Prisma-Cloud-Role等)であれば、redlock=Prisma Cloudの直接証拠となる。ARNとロール名は報告書に転記する。IAMはグローバルサービスのため、どこか1リージョンで1回実行すれば足りる。

## ③ Prismaロールの作成日確認(IAM・1回のみ・保存不要)

**何のコマンドか**: IAMロール一覧から、Prisma関連ロールの作成日(CreateDate)を取得する。

```bash
aws iam list-roles --query \
 "Roles[?contains(RoleName,'risma') || contains(RoleName,'edlock')].[RoleName,CreateDate]" \
 --output text
```

名前でヒットしない場合は、②で判明したロール名で `contains` を書き換えるか、作成日で絞り込む:

```bash
aws iam list-roles --query \
 'Roles[?CreateDate>=`2026-04-19`&&CreateDate<`2026-04-23`].[RoleName,CreateDate]' \
 --output text
```

**結果で何がわかるか**: ロールのCreateDateが2026-04-20〜21であれば、「担当者回答(4/20-21オンボード)」「費用の段差(4/20)」「redlock初出(4/20)」と合わせて四点一致となる。IAMのロール情報はCloudTrailの90日制限と無関係に残り続けるため、ログが遡れなくなった後も参照できる恒久的な証拠になる。

**オプション**: 500アカウント(Hub検証環境)側でも同コマンドを実行し「Prismaロールが存在しない」ことを確認すると、「508のみ費用増加」の対比証拠として補強できる。

## ④ redlockイベント全量のTSV保存(証拠ログ・2ファイル)

**何のコマンドか**: 対象24時間のredlockイベント全件を、東京とus-east-1の2リージョンでTSVに保存する。本手順で唯一ファイル出力するコマンド。

```bash
for R in ap-northeast-1 us-east-1; do
  aws cloudtrail lookup-events --region $R \
    --lookup-attributes AttributeKey=Username,AttributeValue=redlock \
    --start-time $START --end-time $END \
    --query 'Events[].[EventTime,EventName,EventSource,Username]' \
    --output text > redlock_${R}_20260721.tsv
  echo -e "$R\tredlock=$(wc -l < redlock_${R}_20260721.tsv)"
done
```

**結果で何がわかるか**: 2つ。(1) 行数がそのままredlock件数になるため、①の総件数と割り算して直近のredlock比率が出る。リージョンごとに9割前後なら「5月以降・直近も同一要因が継続」と最終報告に記載できる。(2) EventName/EventSource列から、呼び出しがDescribe/List/Get系の読み取りAPIを多サービス横断で約1時間周期に巡回するパターンであることが確認でき、CSPMスキャンの挙動そのものである裏付けになる(4月データの表6と同じ形式なので、4月 vs 直近の対比表が作れる)。

**所要時間の目安**: lookup-eventsのレート制限(毎秒2リクエスト、1ページ50件)のため、①④合わせて15〜20分程度。途中でエラーになった場合はSTART/ENDを12時間ずつ(00:00〜12:00、12:00〜24:00)に分けて実行し、TSVは `cat` で連結する。

## なぜこの範囲で十分か

立証すべき論点ごとに、証拠がどこで揃っているかを対応させると次のとおり。今回の①〜④は「未取得の2論点」だけをピンポイントで埋める設計になっている。

| 論点 | 証拠 | 状態 |
|---|---|---|
| 費用増の費目・規模・時期(4/20段差、10倍、5-6月継続) | Cost Explorerデータ(表1〜5)。全リージョン・全日をカバー | 取得済み |
| 増加開始時点の主体が redlock(約9割) | 4月lookup-events(表6、TOP9リージョン) | 取得済み |
| 508のみ増加した理由 | Prisma導入対象が508のみ(担当者回答)+500は費用横ばい | 取得済み |
| redlock = Prisma Cloud の直接証拠 | ②ロールARN+③作成日4/20-21 | 今回取得 |
| 直近も同一要因が継続 | ①④の比率(直近24時間) | 今回取得 |

**全リージョン×全期間の再取得が不要な理由**: 「件数がどれだけ増えたか」はコストデータ側で既に全リージョン・全日にわたり定量済みであり、lookup-eventsに求める役割は「その件数を誰が発生させているか」の確認だけである。主体の確認はサンプルで統計的に足り、全量を取っても結論は変わらず情報が増えない。加えてレート制限により全リージョン×全量は数時間かかり現実的でない。

**東京+us-east-1の2リージョンで足りる理由**: コストデータで増分が全リージョンほぼ均等(=単一の全リージョン巡回プロセス)であることが確認済みのため、代表リージョンでの比率確認で全体を推せる。その代表として、増分最大で4月データ(表6)と直接対比できるap-northeast-1と、Trail基幹リージョンで同じく4月データがあるus-east-1を選定した。両者で同じ比率が出れば「特定リージョン固有の事象ではない」ことも同時に示せる。

**24時間窓で足りる理由**: Prismaのスキャンは約1時間周期の定常ポーリングであり、24時間で全周期を約24回分含む。日次件数が4/20以降ほぼ一定(コストデータ)であることと合わせ、任意の1日が全期間の代表になる。

## 注意事項

lookup-eventsは90日しか遡れず、本日時点で4/23以前は再取得不可。取得済みの4月TSV(guardduty_selected-regions_2026-04_summary.tsv 等)は必ず保管すること。今回取得する②③のIAM情報は期限の影響を受けないため、最終報告の恒久証拠として優先的に記載する。
